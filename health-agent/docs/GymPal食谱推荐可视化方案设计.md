# GymPal Agent饮食建议可视化定制实施方案

# 一、方案总则

## 1.1 核心目标

依托GymPal现有web页面（对话、概览、计划、档案、记录、动作），实现“Agent饮食建议→智能生成可视化”的闭环，核心呈现“盘子式总体可视化+弹窗卡片详情”效果，让用户直观感知今日饮食比例、食材搭配，点击即可查看完整食谱、三餐分配、热量分布等细节，贴合健身饮食需求，同时与现有页面风格统一、交互流畅，提升用户使用体验和饮食建议的落地性。

## 1.2 适配现有页面说明

结合应用现有页面布局，明确可视化模块的嵌入位置，确保不破坏原有页面逻辑，实现无缝衔接：

- 概览（仪表盘）：核心嵌入位置，放置“盘子式总体可视化”组件，作为用户进入应用后查看饮食建议的首要入口，与运动数据、今日计划等模块并列，形成“运动+饮食”的全景概览；

- 对话页面：与Agent对话获取饮食建议后，自动同步更新概览页面的盘子可视化效果，并弹出简易提示“饮食建议已更新，可前往概览查看详情”，支持点击提示直接跳转至概览；

- 计划页面：可将饮食建议可视化卡片嵌入“今日计划”模块，与运动计划并列展示，方便用户同步查看当日运动与饮食安排；

- 记录页面：用户记录实际饮食后，可对比“盘子式可视化建议”与“实际饮食”的差异，同步更新热量、营养比例的对比数据；

- 档案（profile）页面：结合用户档案中的运动目标（减脂/增肌/塑形）、饮食禁忌，动态调整盘子可视化的食材比例和弹窗详情内容，实现个性化适配；

- 动作页面：暂不嵌入饮食可视化模块，避免功能混淆，仅在用户完成运动后，通过系统提示引导至概览页面查看适配的饮食建议。

## 1.3 核心原则

- 视觉统一：贴合GymPal现有web应用的UI风格（色彩、字体、圆角等），盘子可视化和弹窗卡片的设计不突兀，保持整体视觉连贯性；

- 便捷可操作：点击盘子即可弹出详情卡片，操作步骤≤1步，弹窗可快速关闭、拖拽，不干扰用户其他操作；

- 精准适配：可视化内容完全同步Agent的饮食建议，食材比例、热量、食谱等数据实时联动，无偏差；

- 灵活可选：支持卡通、逼真两种盘子视觉风格，用户可在设置中切换，适配不同审美偏好；

- 可落地性：技术方案贴合web开发常规流程，选用成熟插件和素材，降低开发成本，确保快速落地。

# 二、视觉呈现设计（核心落地细节）

## 2.1 总体可视化：盘子式呈现（核心组件）

核心效果：一个圆形盘子（直径适配web端概览页面，建议宽度280-320px），内部按Agent建议的营养比例（蛋白质、碳水化合物、脂肪）划分区域，每个区域放置对应食材的卡通/逼真图标，直观展示今日饮食的食材搭配和比例分布，贴合哈佛健康饮食餐盘的分区逻辑，让抽象的营养比例转化为直观的视觉呈现，无需用户记忆复杂数字。

### 2.1.1 盘子基础设计（两种风格，可切换）

- 风格1：卡通风格（默认）
        

    - 盘子样式：简约卡通圆形，边框圆润（圆角8px），底色为米白色，边缘添加浅灰色描边（1px），整体风格活泼、清新，贴合健身App的活力氛围；

    - 食材呈现：采用Q版卡通食材图标（如卡通鸡胸肉、米饭、西兰花、鸡蛋等），色彩鲜艳、辨识度高，每个食材图标大小与对应营养比例匹配（比例越高，图标越大/数量越多）；

    - 比例标注：盘子内部每个营养区域添加简约文字标注（如“蛋白质25%”“碳水50%”），字体简洁（建议14px，无衬线字体），颜色与对应区域食材图标颜色呼应（如蛋白质区域用浅蓝色、碳水用浅绿色），不遮挡食材图标；

    - 交互提示：盘子右下角添加小型“点击查看详情”图标（问号/箭头），鼠标悬浮时显示提示文字，引导用户点击。

- 风格2：逼真风格（可选）


    - 盘子样式：模拟真实陶瓷餐盘，有轻微纹理和光泽，底色为纯白色，边缘有自然的浅灰色阴影，增强立体感；

    - 食材呈现：采用高清实拍食材图片（如真实鸡胸肉、全麦面包、新鲜蔬菜等），按比例裁剪拼接，贴合真实餐盘摆盘效果，避免模糊、失真，参考视觉中国等素材网站的健身餐实拍风格；

    - 比例标注：文字标注隐藏，鼠标悬浮在对应区域时，弹出半透明提示框，显示“碳水50%（米饭、红薯）”等信息，不破坏逼真视觉效果；

    - 交互提示：鼠标悬浮在盘子上时，光标变为“手型”，盘子轻微放大（缩放比例1.05），提示可点击。

### 2.1.2 比例适配逻辑

盘子分区比例完全同步Agent输出的饮食建议，核心逻辑参考哈佛健康饮食餐盘的“半盘蔬果、四分之一蛋白、四分之一主食”原则，结合健身人群需求调整，具体对应关系：

- 碳水化合物（主食类）：占比40%-60%（根据用户目标调整，增肌偏高、减脂偏低），对应区域放置米饭、红薯、全麦面包等食材；

- 蛋白质（肉蛋乳类）：占比20%-30%，对应区域放置鸡胸肉、鸡蛋、虾仁、牛奶等食材；

- 脂肪（健康脂肪类）：占比10%-20%，对应区域放置坚果、牛油果、橄榄油等食材；

- 蔬菜/水果：融入对应营养区域（如蔬菜计入碳水、水果计入碳水/脂肪），确保视觉上的丰富度，同时贴合“多色原则”，搭配不同颜色的蔬果，提升视觉吸引力。

示例：减脂用户Agent建议碳水50%、蛋白质30%、脂肪20%，则盘子一半区域为碳水类食材（米饭、西兰花），30%为蛋白质类（鸡胸肉、鸡蛋），20%为脂肪类（坚果、牛油果），直观呈现均衡饮食结构。

### 2.1.3 页面嵌入细节

- 概览页面：放置在“今日概览”模块的右侧，与“今日运动目标”“已完成运动”模块并列，占据页面1/3宽度，确保不挤压其他核心内容；

- 响应式适配：电脑端（屏幕宽度≥1200px）显示完整盘子+比例标注；平板端（768px≤宽度＜1200px）缩小盘子尺寸（宽度220-260px），简化标注；手机端（宽度＜768px）点击“饮食建议”图标，弹出盘子可视化弹窗，避免页面拥挤；

- 动态效果：Agent更新饮食建议后，盘子区域有轻微的“刷新动画”（旋转10°后恢复），提示用户内容已更新；用户切换风格时，有平滑的过渡动画（淡入淡出，时长0.3秒）。

## 2.2 详情弹窗卡片（点击触发）

核心效果：点击盘子任意区域，弹出一个悬浮卡片（居中显示，可拖拽），卡片内完整展示Agent推荐的饮食详情，包含食谱、三餐分配、热量分布等，布局清晰、重点突出，支持快速关闭、收藏、分享。

### 2.2.1 卡片基础设计

- 样式：长方形卡片（宽度600px，高度450px，适配电脑端；手机端宽度为屏幕80%，高度自适应），圆角12px，背景为白色，添加轻微阴影（模糊度8px），避免与页面背景混淆；

- 顶部：标题栏（高度50px），左侧显示“今日饮食建议”，右侧为“关闭”“收藏”“分享”三个图标（简洁线条图标），点击关闭可关闭卡片，收藏后同步至“计划”页面，分享支持复制链接、分享至社交平台；

- 内容布局：卡片内部按“总览-三餐详情-营养明细”分区，采用纵向滚动（内容超出高度时），滚动流畅无卡顿；

- 视觉风格：与盘子可视化风格一致（卡通/逼真），卡片内食材图片、图标与盘子保持统一，字体、颜色与GymPal现有页面一致，避免视觉割裂。

### 2.2.2 卡片内容详情（核心必含）

1. 今日饮食总览（顶部区域，占比20%）
        

    - 核心数据：今日目标热量、推荐总热量、热量缺口/盈余（结合用户运动消耗），用简约数字+文字标注（如“目标热量：1800千卡 | 推荐摄入：1600千卡 | 缺口：200千卡”）；

    - 营养比例：小型环形图（直径80px），展示蛋白质、碳水、脂肪的具体占比和数值（如“蛋白质：48g（30%）”），与盘子比例一致，参考ECharts自定义图表的简洁风格，避免复杂刻度；

    - 适配提示：简短文字提示（如“减脂期·低卡高纤，适配今日1小时有氧训练”），贴合用户运动目标和当日运动计划。

2. 三餐饮食详情（中间区域，占比60%）
        

    - 分“早餐、午餐、晚餐”三个标签页，可点击切换，默认显示全部三餐；

    - 每顿饭包含：食材名称、分量（克）、烹饪方式、热量、核心营养（如“鸡胸肉：120g，水煮，130千卡，蛋白质25g”）；

    - 呈现形式：卡通风格用卡通食材图标+文字，逼真风格用实拍食材小图+文字，每顿饭底部标注“单餐总热量”，方便用户快速查看；

    - 灵活调整：每道食材右侧添加“替换”按钮，点击可弹出Agent推荐的同类替换食材（如鸡胸肉替换为虾仁），替换后实时更新卡片内热量、营养数据，同步更新盘子可视化比例。

3. 营养明细与备注（底部区域，占比20%）
        

    - 营养明细：表格形式展示蛋白质、碳水、脂肪、膳食纤维的“推荐摄入量”“实际推荐量”“剩余可摄入”，清晰呈现营养是否达标；

    - Agent备注：Agent给出的个性化饮食提示（如“运动后30分钟内食用晚餐，有助于肌肉修复”“避免油炸烹饪，减少热量摄入”）；

    - 记录入口：底部添加“记录今日饮食”按钮，点击可跳转至“记录”页面，自动填充推荐的饮食内容，用户可手动修改实际食用情况。

### 2.2.3 弹窗交互细节

- 触发方式：点击盘子任意区域、盘子右下角提示图标，均可触发弹窗；

- 关闭方式：点击右上角“关闭”图标、点击卡片外部空白区域、按键盘Esc键，均可关闭弹窗；

- 拖拽功能：点击卡片标题栏，可拖拽卡片至页面任意位置，避免遮挡页面核心内容（如运动计划、对话窗口）；

- 动态效果：弹窗弹出时，有“淡入+轻微上移”动画（时长0.3秒）；关闭时，有“淡出+轻微下移”动画，提升交互质感；切换三餐标签页时，有平滑的滑动过渡。

## 2.3 视觉风格统一规范

- 色彩：沿用GymPal现有主色调（如活力橙、简约白、沉稳灰），盘子和卡片的色彩不新增额外色系，避免视觉杂乱；营养区域颜色统一（碳水=浅绿色、蛋白质=浅蓝色、脂肪=浅橙色）；

- 字体：统一使用无衬线字体（如Inter、微软雅黑），标题18px（加粗）、正文14px、备注12px，确保清晰易读，避免使用复杂字体；

- 图标：所有图标（关闭、收藏、分享、食材图标等）风格统一，卡通风格用圆润线条，逼真风格用简约写实图标，避免混用；

- 间距：卡片内部、盘子内部的间距统一（内边距16px，食材间距8px），与现有页面的间距规范保持一致，提升整体协调性。

# 三、技术落地方案（可直接对接开发）

## 3.1 开发前提（Agent数据标准化）

Agent输出饮食建议时，需按以下标准化格式输出，确保可视化模块可直接调用，避免数据混乱，对接后端开发接口：

```json
{
  "userGoal": "减脂", // 用户运动目标（减脂/增肌/塑形）
  "totalCalorie": 1600, // 今日推荐总热量（千卡）
  "targetCalorie": 1800, // 今日目标热量（千卡）
  "nutritionRatio": { // 营养比例（对应盘子分区）
    "carbohydrate": 50, // 碳水占比（%）
    "protein": 30, // 蛋白质占比（%）
    "fat": 20 // 脂肪占比（%）
  },
  "nutritionDetail": { // 营养明细（对应卡片底部表格）
    "protein": { "target": 50, "recommend": 48, "remaining": 48 },
    "carbohydrate": { "target": 225, "recommend": 200, "remaining": 200 },
    "fat": { "target": 40, "recommend": 36, "remaining": 36 },
    "fiber": { "target": 25, "recommend": 22, "remaining": 22 }
  },
  "meals": [ // 三餐详情（对应卡片中间区域）
    {
      "mealType": "breakfast", // 早餐/午餐/晚餐（breakfast/lunch/dinner）
      "totalCalorie": 480, // 单餐总热量
      "foods": [
        {
          "name": "全麦面包", // 食材名称
          "weight": 80, // 分量（克）
          "calorie": 240, // 热量（千卡）
          "nutrition": { "protein": 8, "carbohydrate": 40, "fat": 4 }, // 单食材营养
          "cooking": "无油", // 烹饪方式
          "replaceable": [{"name": "燕麦", "weight": 50, "calorie": 180}] // 可替换食材
        },
        // 其他食材...
      ]
    },
    // 午餐、晚餐...
  ],
  "remark": "运动后30分钟内食用晚餐，有助于肌肉修复", // Agent备注
  "fitTips": "减脂期避免油炸烹饪，减少热量摄入" // 适配提示
}
}
```

## 3.2 前端开发（核心实现）

### 3.2.1 技术选型

- 核心框架：沿用GymPal现有web框架（如Vue3、React），确保代码兼容性，减少开发成本；

- 可视化插件：
        

    - 盘子可视化：采用ECharts或D3.js实现，参考ECharts自定义仪表盘、D3 Pie饼图的开发思路，绘制圆形盘子及营养分区，嵌入食材图标/图片，实现比例精准呈现；ECharts可通过自定义series配置，实现盘子的分区渲染和食材图标嵌入，D3.js可实现更灵活的自定义摆盘效果；

    - 卡片内环形图：使用ECharts简约环形图，隐藏多余刻度，仅展示核心营养比例和数值，参考ECharts gauge图表的简化配置；

- 弹窗组件：使用现有UI组件库（如Element Plus、Ant Design）的弹窗组件，自定义内容布局，添加拖拽功能（可使用vue-draggable、react-draggable插件）；

- 响应式适配：使用CSS媒体查询，适配不同屏幕尺寸，调整盘子大小、卡片布局、字体大小；

- 动画效果：使用CSS3动画（transition、transform）实现盘子刷新、弹窗淡入淡出、标签页切换等动态效果，避免使用复杂动画导致卡顿。

### 3.2.2 核心功能开发步骤

1. 第一步：盘子可视化组件开发（3个工作日）
        

    - 基于ECharts/D3.js，绘制圆形盘子，按营养比例划分区域，设置两种风格（卡通/逼真）的样式；

    - 嵌入食材图标/图片，根据Agent输出的食材列表，匹配对应区域，调整图标大小与比例适配；

    - 添加比例标注、交互提示、动态刷新动画，实现鼠标悬浮提示、点击触发弹窗功能；

    - 开发风格切换功能，用户在设置页面切换风格后，实时更新盘子样式，同步更新卡片内食材呈现风格。

2. 第二步：弹窗卡片组件开发（4个工作日）
        

    - 基于现有UI组件库，开发悬浮弹窗卡片，实现关闭、收藏、分享、拖拽功能；

    - 开发卡片内“总览-三餐详情-营养明细”布局，对接Agent数据接口，渲染热量、营养、食谱等数据；

    - 实现三餐标签页切换、食材替换功能，替换食材后，实时更新卡片数据和盘子可视化比例；

    - 添加“记录今日饮食”按钮，对接记录页面接口，实现数据同步。

3. 第三步：页面嵌入与联动开发（3个工作日）

    - 将盘子可视化组件嵌入概览页面，调整布局，确保与现有模块适配；

    - 实现对话页面与盘子组件的联动：用户与Agent对话获取饮食建议后，触发盘子组件刷新，弹出提示并支持跳转；

    - 实现与记录页面的联动：用户在卡片点击“记录饮食”，跳转至记录页面并自动填充数据；用户在记录页面修改实际饮食后，同步更新卡片内“剩余可摄入”数据；

    - 实现与档案页面的联动：根据用户档案中的目标、禁忌，调整盘子比例和食材推荐（如素食用户替换肉类食材为豆制品）。

4. 第四步：响应式适配与兼容性测试（2个工作日）
        

    - 适配电脑端、平板端、手机端，调整盘子大小、卡片布局、字体大小，确保不同设备上视觉清晰、操作便捷；

    - 测试数据同步稳定性，确保Agent更新建议后，盘子和卡片数据实时同步，无延迟、无偏差。

## 3.3 后端开发（数据支撑）

- 接口开发：
        

    - Agent饮食建议输出接口：按标准化格式输出饮食建议数据，供前端可视化组件调用；

    - 可视化数据同步接口：实现前端与后端的数据同步，Agent更新建议后，实时推送数据至前端，触发盘子和卡片刷新；

    - 食材替换接口：用户在卡片点击“替换食材”时，后端返回Agent推荐的同类食材数据，确保替换后营养比例、热量数据准确；

    - 收藏/分享接口：实现卡片收藏功能（同步至用户计划）、分享功能（生成链接/图片）。

- 数据存储：
        

    - 存储用户每日饮食建议数据，关联用户ID、日期、运动目标，支持历史查询（用户可查看过往饮食建议）；

    - 存储用户风格偏好（卡通/逼真）、食材替换记录，下次生成建议时，优先适配用户偏好；

    - 关联用户运动数据、记录数据，确保热量缺口/盈余计算准确，实现“运动+饮食”数据联动。

- 数据校验：添加数据校验机制，确保Agent输出的数据格式正确、营养比例合理，避免前端渲染错误；若数据异常，返回默认饮食建议，同时提示后端排查问题。

## 3.4 Agent优化（提升适配性）

- 数据输出优化：确保Agent输出的饮食建议符合前端可视化的标准化格式，包含盘子分区所需的营养比例、食材列表，以及卡片所需的三餐详情、营养明细；

- 个性化适配优化：Agent结合用户档案中的运动目标、饮食禁忌、口味偏好，生成更精准的食材推荐和比例分配，确保盘子和卡片内容贴合用户需求（如乳糖不耐受用户，替换牛奶为无糖豆浆）；

- 交互联动优化：Agent与前端可视化组件联动，用户在对话中修改饮食需求（如“减少碳水”“替换鸡胸肉为鱼肉”），Agent实时更新建议，前端同步刷新盘子和卡片数据，无需用户手动刷新页面。

# 七、风险与应对措施

- 风险1：Agent输出数据不标准，导致盘子和卡片渲染错误；应对：提前制定严格的数据输出规范，开发前进行接口联调，添加数据校验机制，若数据异常，返回默认饮食建议，同时提示后端排查问题，参考ECharts图表的数据校验逻辑，避免渲染失败；

- 风险2：盘子可视化比例与Agent建议不一致，影响用户体验；应对：前端开发时，添加比例校验逻辑，确保盘子分区比例与Agent输出的营养比例完全匹配，每一次数据更新后，自动校验比例准确性，出现偏差时及时修正；

- 风险3：弹窗卡片拖拽时卡顿、遮挡核心内容；应对：优化拖拽功能，减少拖拽时的资源占用，添加“自动避让”逻辑，拖拽时避开页面核心模块（如对话窗口、运动计划），避免遮挡；

- 风险4：素材版权问题；应对：所有素材均选择免费可商用或已获得授权的资源，建立素材版权台账，定期排查，避免侵权；

- 风险5：不同设备上视觉呈现不一致；应对：加强响应式适配测试，针对不同屏幕尺寸，制定详细的视觉规范，确保盘子、卡片的样式、字体、间距在所有设备上保持一致。
